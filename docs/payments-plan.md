# תוכנית סליקה — תשלום אמיתי באתר

מצב היום: הזמנות נשלחות בוואטסאפ ונרשמות ב־DB (`orders` collection, ערוץ
"whatsapp"). בעגלה יש כבר כפתור "💳 תשלום מאובטח באתר — בקרוב" מנוטרל.

## ספקים מומלצים לעסק קטן בישראל

| ספק | יתרונות | חסרונות |
|---|---|---|
| **Grow (משולם)** | פופולרי בעסקים קטנים, דף תשלום מתארח (אין PCI), עמלה פר עסקה בלי דמי חודש בסיסיים, תומך ביט/אפל פיי | API פשוט אך תיעוד בינוני |
| **PayPlus** | API מסודר, דף תשלום מתארח, תומך הוראות קבע | דמי חודש |
| **Cardcom / Tranzila** | ותיקים, נפוצים | ממשקים מיושנים יותר |
| **PayPal** | קל לחבר | עמלות גבוהות, פחות נפוץ בקניות בארץ |

**המלצה: Grow (משולם)** — הכי קל לפתוח, מתאים להיקפים של חנות שכונתית,
ויש דף תשלום מתארח כך שפרטי אשראי לעולם לא עוברים דרך השרת שלנו.

## איך זה יתחבר (כשיהיה חשבון + API key)

1. **Backend** — endpoint חדש `POST /pay/create`:
   - מקבל את העגלה, יוצר `Order` עם `status: "pending-payment"`, `channel: "card"`.
   - קורא ל־API של הספק ליצירת עסקה ומחזיר ללקוח את כתובת דף התשלום.
2. **דף תשלום מתארח** — הלקוח מועבר לדף של הספק (מאובטח, מותאם מובייל).
3. **Webhook** — `POST /pay/callback`: הספק מודיע על הצלחה/כישלון,
   ה־Order מתעדכן ל־`paid` / `failed`, והדשבורד מציג את זה בלשונית ההזמנות.
4. **Frontend** — הכפתור המנוטרל הופך לאמיתי: לחיצה → `/pay/create` → redirect.
5. דרישות צד־עסק: אישור עוסק/חברה, חשבון בנק, וזהו — אין צורך ב־PCI כי
   משתמשים בדף מתארח.

הערכת עבודה כשמתקבל ה־API key: יום־יומיים כולל בדיקות.
